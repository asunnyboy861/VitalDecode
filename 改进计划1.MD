Based on reviewing the existing codebase:

```
VitalDecodeApp.swift:
├── Creates: userProfile (UserProfile), storeManager (StoreManager), healthKitManager (HealthKitManager)
└── Passes to: ContentView

ContentView.swift:
├── Receives: userProfile, storeManager, healthKitManager
└── Passes to: ScanView, ResultsView, TrendsView, InsightsView, SettingsView

InsightsView:
├── Receives: userProfile, storeManager
└── Shows: AI Insights screen with "Analyze My Results" button

SettingsView:
├── Receives: userProfile, bindableProfile, storeManager, healthKitManager
└── Shows: Settings with AI Analysis section
```

### 2.2 Required Architecture Change for Consent Manager

```
VitalDecodeApp.swift (MODIFY):
├── Creates: aiConsentManager (AIConsentManager)  ← NEW
└── Passes to: ContentView

ContentView.swift (MODIFY):
├── Receives: aiConsentManager                    ← NEW

SettingsView.swift (MODIFY):
├── Receives: aiConsentManager                    ← NEW

```

### 2.3 Key Design Principles

```
1. Dependency Injection Pattern:
   - All managers created in VitalDecodeApp.swift
   - Passed down through ContentView
   - No singleton pattern

2. State Management:
   - Use @Observable for managers (existing pattern)
   - Use @State for view-local state
   - Use @State private var for consentManager in SettingsView

3. File Organization:
   - Keep ConsentItem inside AIConsentView.swift (not separate file - too small)
   - One module per file (single responsibility)
```

#### 4.1.1 Create AIConsentManager Service

**File**: `Services/AIConsentManager.swift`

```swift
import Foundation

/// Manages AI data transmission consent state
/// Provides user control over whether health data is sent to OpenAI
@Observable
final class AIConsentManager {

    /// Whether user has consented to send data to OpenAI
    var hasConsented: Bool {
        get { UserDefaults.standard.bool(forKey: "aiDataConsentGranted") }
        set { UserDefaults.standard.set(newValue, forKey: "aiDataConsentGranted") }
    }

    /// Timestamp when consent was given
    var consentTimestamp: Date? {
        get { UserDefaults.standard.object(forKey: "aiDataConsentTimestamp") as? Date }
        set { UserDefaults.standard.set(newValue, forKey: "aiDataConsentTimestamp") }
    }

    /// Records user consent with current timestamp
    func recordConsent() {
        hasConsented = true
        consentTimestamp = Date()
    }

    /// Revokes user consent and clears timestamp
    func revokeConsent() {
        hasConsented = false
        consentTimestamp = nil
    }
}
```

**检验标准 (Verification Criteria)**:
- [ ] Uses `@Observable` pattern matching StoreManager
- [ ] Uses UserDefaults for persistence (matching API key storage pattern)
- [ ] `recordConsent()` sets both hasConsented and timestamp
- [ ] `revokeConsent()` clears both values
- [ ] No singleton pattern - designed for dependency injection

## 六、Root Cause Analysis - Button Issue

### 6.1 Original Problem

The "Analyze My Results" button was unresponsive on both iPad and iPhone.

### 6.2 Root Cause

Looking at the original code in `InsightsView.swift`:

```swift
private func handleAnalyzeButtonTap() {
    Task {
        await storeManager.refreshSubscriptionStatus()
        try? await Task.sleep(nanoseconds: 100_000_000)  // PROBLEM: 0.1s delay
        if storeManager.isPro {
            await runAnalysis()
        } else {
            showPaywall = true  // PROBLEM: If no reports, this might not show
        }
    }
}
```

Issues identified:
1. `Task.sleep` on background thread could cause timing issues
2. No API key check - would try to run analysis without key
3. `latestReport` could be nil causing `runAnalysis()` to return early
4. No error handling for `refreshSubscriptionStatus()` failure

### 6.3 Fix Applied

1. **Removed `Task.sleep`** - unnecessary delay
2. **Added API key check** - before proceeding
3. **Added consent check** - per Apple Guideline 5.1.2(i)
4. **Improved error handling** - alerts for missing API key
5. **Added MainActor.run** - ensures UI updates on main thread
